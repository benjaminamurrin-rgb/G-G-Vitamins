Part 1 — Files, Folders, and Paths

Everything in computing sits on top of a simple idea: information is stored in files, and files are organised into folders. Get comfortable with this and half the jargon stops being scary.

The building blocks

Bit and byte. A bit is the smallest unit of digital information: a single 0 or 1. Eight bits make a byte, which is roughly enough to store one character of text (the letter "A", say). Bytes are the unit everything else is measured in: a kilobyte (KB) is about a thousand bytes, a megabyte (MB) about a million, a gigabyte (GB) about a billion. You do not need to think in bits day-to-day, but this is where "file size" comes from.

File. A file is a single named lump of stored information — a document, a photo, a piece of code, a song. Every file has a name and, usually, an extension.

File extension. The bit after the last dot in a filename, such as .txt, .pdf, .png, .py, or .html. It signals what kind of file it is and which program should open it. The extension is just a naming convention — renaming photo.png to photo.txt does not change the actual contents, it just misleads programs about how to read them.

Folder / directory. A container that holds files and other folders. "Folder" and directory mean exactly the same thing — "folder" is the friendly word used in graphical interfaces, "directory" is the word used on the command line and by programmers. A folder inside another folder is a subdirectory.

The filesystem: a tree

Filesystem. The overall structure that organises every file and folder on a computer. It is shaped like an upside-down tree: one starting point at the top, branching into folders, branching into more folders and files. This tree shape is why we talk about a folder's "parent" and "children".

C:\                          <- the "root" on Windows (top of the tree)
└─ Users
   └─ Murrin Benjamin        <- your "home" folder
      ├─ Documents
      │   ├─ report.docx      <- a file
      │   └─ budget.xlsx
      ├─ Pictures
      │   └─ holiday.png
      └─ Claude Projects      <- a folder holding a project
          ├─ script.py
          └─ notes.txt

Root. The very top of the filesystem tree — the folder that contains everything else. On Windows it is a drive letter such as C:\. On Mac and Linux it is written as a single forward slash /. (Confusingly, "root" is also the name of the all-powerful administrator account on Mac/Linux — see Part 12. Same word, two meanings.)

Home directory. Your personal folder, where your own files live by default. On Windows it is typically C:\Users\YourName. It is often abbreviated with a tilde: ~.

Path: the address of a file

Path. A path is the written address that tells the computer exactly where a file or folder is in the tree. You build it by listing the folders you pass through, separated by slashes. (Windows traditionally uses backslashes \; almost everything else uses forward slashes /.)

There are two kinds of path, and the distinction matters constantly:

Absolute path. The full address, starting from the root. It is unambiguous from anywhere. Example: C:\Users\Murrin Benjamin\Claude Projects\script.py. Like giving your full postal address including country.

Relative path. An address given relative to wherever you currently are. If you are already sitting in the Claude Projects folder, the relative path to the same file is just script.py. Like saying "two doors down on the left" — only meaningful if the listener knows where you are standing.

Two special shorthands you will see everywhere in paths:

   .     means "the current folder I am in"
   ..    means "the parent folder, one level up"

Example, if you are in  .../Claude Projects/  :
   ./notes.txt          -> notes.txt in this folder
   ../Pictures/x.png    -> go up to "Murrin Benjamin", then into Pictures

Working directory (a.k.a. current directory). The folder a program considers itself to be "in" right now. Relative paths are interpreted from here. When you open a terminal (Part 2) or when Claude Code runs, it always has a working directory — the folder it treats as home base for the task.

What is inside a file

Plain text vs binary. A plain-text file contains nothing but readable characters — letters, numbers, punctuation. You can open it in any basic editor and understand it. Code files (.py, .html, .txt, .csv, .json) are plain text. A binary file stores information in a compact machine format that looks like gibberish if you open it as text — images, videos, PDFs, and programs are binary. The distinction matters because most programming tools (and Git, and Claude Code) work most naturally with plain text.

Text editor. A program for creating and changing plain-text files. This ranges from very simple (Notepad) to very powerful (VS Code — see Part 12). A text editor is not a word processor: it does not do fonts and page layout, it deals in raw characters, which is exactly what you want for code.

Part 2 — Talking to the Computer: the Command Line

You already know one way to control a computer: clicking on things. There is a second way — typing commands — that is more powerful for many tasks and is the native environment of programming and of Claude Code.

Two ways to control a machine

TermWhat it isEveryday analogy
GUI
(Graphical User Interface)		Controlling the computer by pointing and clicking on windows, icons, and buttons. Pronounced "gooey". This is the desktop you already use.Ordering from a picture menu — easy to browse, limited to what is shown.

(Command-Line Interface)		Controlling the computer by typing text commands, one line at a time. Faster, scriptable, and able to do things no button exists for.Ordering by writing exactly what you want — harder to learn, but unlimited.

CLI. As above: a way of operating a program or a whole computer by typing instructions rather than clicking. Most professional developer tools — including Git, Python, npm, and Claude Code — are used primarily through a CLI. Learning to read and type commands is the single biggest unlock in this whole document.

Where you type commands

Terminal. The window or application in which you type commands and read their text output. On Windows this might be "Windows Terminal", "PowerShell", or "Command Prompt"; on Mac it is "Terminal". The terminal is just the container — the panel where the conversation happens.

Shell. The actual program running inside the terminal that reads your typed command, carries it out, and prints the result. Common shells are Bash and Zsh (Mac/Linux) and PowerShell (Windows). Think of it this way: the terminal is the telephone handset; the shell is the person on the other end who understands what you say and acts on it. In casual speech people say "the terminal", "the command line", and "the shell" interchangeably.

Prompt (command-line sense). The little bit of text the shell prints to say "I'm ready for your next command", often ending in $, >, or #. Do not confuse this with an AI "prompt" (Part 3) — same word, different worlds.

The anatomy of a command

   git   commit   -m   "fix the login bug"
   ───   ──────   ──   ──────────────────
    │       │      │            │
 command   sub-  flag/       argument
          command option   (a value the
                            command uses)

Command. The name of the program you want to run (here, git).
Argument. A piece of input you hand to the command — often a filename or a value it should act on.
Flag / option / switch. A short modifier, usually starting with one dash - or two --, that changes how the command behaves. -m above means "the next thing is the message". Flags are why the same command can do many different things.

A handful of commands worth recognising

CommandWhat it does
pwd"Print working directory" — shows which folder you are currently in.
ls (or dir)Lists the files and folders in the current directory.
cd foldername"Change directory" — moves you into a different folder. cd .. goes up one level.
mkdir name"Make directory" — creates a new folder.
cat filePrints the contents of a text file to the screen.
cp / mv / rmCopy / move (or rename) / remove a file. rm deletes — there is no recycle bin on the command line, so it is permanent. Handle with care.

Standard output and standard error. When a command runs, its normal results are printed to a stream called standard output (stdout); anything that went wrong is printed to a separate stream called standard error (stderr). They usually both appear in your terminal, but keeping them separate lets tools capture results and errors independently.

Script. A plain-text file containing a list of commands, saved so you can run them all at once instead of typing them one by one. A shell script automates a sequence of terminal commands; a Python script (Part 5) is a file of Python instructions. "Scripting" simply means automating a task by writing it down as code.

Part 3 — What AI Actually Is

You know AI can write and answer questions. This part gives you the vocabulary for how, so the rest of the document (and your daily use of Claude) makes sense.

The family tree of the field

Artificial Intelligence (AI). The broad field of building computer systems that perform tasks we would normally call "intelligent" — understanding language, recognising images, making decisions. It is an umbrella term, not one specific technology.

Machine learning (ML). The dominant approach to AI today. Instead of a human writing explicit rules for every situation, the computer learns patterns from examples. Show it millions of examples and it works out the statistical regularities itself.

Neural network. The particular kind of machine-learning system behind modern AI, loosely inspired by how brain cells connect. It is a vast web of simple mathematical units ("neurons") whose connection strengths are tuned during learning. You do not need the maths — just know that a "neural network" is the machinery inside.

Model. The finished, trained neural network — the actual thing that produces answers. When people say "which model are you using?" they mean which specific trained system (see the Claude models below). A model is essentially an enormous file of learned numbers plus the code to run it.

Language models specifically

Language model. A model trained on huge amounts of text whose core skill is predicting what text should come next given the text so far. That sounds narrow, but doing it extremely well — across every topic, style, and language — produces something that can answer questions, write code, summarise, translate, and reason. LLM stands for Large Language Model: "large" because these models have billions of internal values and were trained on a large fraction of the written internet. Claude is an LLM.

Token. Language models do not read whole words or whole letters — they read tokens, which are chunks of text roughly the size of a short word or a word-fragment. "Cat" might be one token; "unbelievable" might be three (un, believ, able). Everything is priced and measured in tokens: how much you can send, how much the model writes, and often what you pay. A rough rule of thumb: 1 token ≈ 4 characters ≈ ¾ of a word in English.

How a language model produces an answer (simplified):

  Your text  ->  split into  ->  model predicts  ->  next token  ->  repeat until
  (a prompt)     TOKENS          most likely          appended       it decides to stop
                                 next token                          ->  full answer

  It writes one token at a time, each choice informed by everything before it.

Prompt (AI sense). The text you give the model to respond to — your question, instruction, or the document you want it to work on. "Prompting" is the craft of phrasing that input to get good results. This is a different meaning from the command-line "prompt" in Part 2.

Context / context window. The context is everything the model can "see" at once for a given task: your prompt, the conversation so far, any files or data included, and its own replies. The context window is the maximum size of that working memory, measured in tokens. When a conversation grows past the window, the oldest material must be dropped or summarised — which is why very long sessions sometimes seem to "forget" early details. Claude's context window is large (hundreds of thousands of tokens), but it is not infinite.

Compaction / summarisation. When context gets close to full, a tool like Claude Code can automatically compress earlier parts of the conversation into a summary to make room, keeping the gist while freeing up tokens. This is why a long Claude Code session can keep going rather than hitting a hard wall.

Training, running, and getting things wrong

Training. The one-time, hugely expensive process of feeding the model massive amounts of text and adjusting its internal numbers until it predicts well. Training produces the finished model. It happened in the past; you are not training Claude when you chat with it.

Inference. The everyday act of running the finished model to get an answer. Every time you send Claude a message, that is inference. (Training = building the engine; inference = driving the car.)

Parameters / weights. The billions of internal numbers a model learned during training. More parameters can mean more capability but also more cost to run. "Weights" is the more technical synonym.

Knowledge cutoff. Because training happened at a fixed point, a model's built-in knowledge stops at a certain date. Anything after that it does not inherently know — which is why tools that fetch live information (web search, reading your files) are so valuable.

Hallucination. When a model states something false or invented with complete confidence. It happens because the model generates plausible-sounding text, and plausible is not the same as true. This is the reason to verify important facts, test generated code, and never assume output is correct just because it reads smoothly.

The Claude model family

Claude is offered in a few sizes, trading speed and cost against raw capability. Names and versions change over time, but the pattern is consistent:

ModelCharacter
OpusThe most capable and thorough — best for hard reasoning, complex code, and deep analysis. Slower and more expensive.
SonnetThe balanced workhorse — very capable, noticeably faster and cheaper. The default for most work.
HaikuThe fastest and cheapest — great for simple, high-volume, or quick tasks where top-tier reasoning is not required.

Part 4 — Claude Code: Agents, Tool Use, and MCP

Ordinary chat AI can only talk. Claude Code can act — on your actual computer. Understanding how turns it from a mysterious black box into a controllable tool.

Chat vs an agent

Claude (the assistant). The AI model you converse with. On its own it produces text.

Claude Code. A program that runs on your machine and gives Claude the ability to do things: read and write your files, run terminal commands, search the web, and more — then observe the results and decide what to do next. It runs in your terminal or code editor and operates inside a working directory (Part 1) that you point it at.

Agent / agentic. An agent is an AI that does not just answer in one shot but works towards a goal over multiple steps: it makes a plan, takes an action, looks at the outcome, and adjusts — looping until the task is done. "Agentic" describes this act-observe-adapt behaviour. Claude Code is an agent. This is why it can carry out a multi-step request like "find the bug and fix it" rather than just describing how you might.

Tool use: how Claude acts on the world

Tool use (also called function calling). This is the central mechanism. A tool is a specific capability the model is allowed to invoke — "read this file", "run this command", "search the web". The model cannot touch your computer directly; instead, when it decides an action is needed, it emits a structured request to use a named tool with specific inputs. Claude Code carries out that request in the real world and hands the result back to the model, which reads it and continues. The loop below is the beating heart of how Claude Code works.

The tool-use loop:

   YOU                 CLAUDE                    A TOOL (on your machine)
    |                    |                              |
    | "fix the bug" ---> |                              |
    |                    | decides: read the file       |
    |                    | Read(app.py) --------------> | (opens the file)
    |                    | <----- file contents -------- |
    |                    | decides: edit line 42        |
    |                    | Edit(app.py) --------------> | (changes the file)
    |                    | <----- "done" --------------- |
    |                    | decides: run the tests       |
    |                    | Bash("pytest") ------------> | (runs the command)
    |                    | <----- "all passed" --------- |
    | <-- "Fixed it" --- |                              |

Permissions. Because tools take real actions, Claude Code asks your approval before anything consequential — running a command, editing a file, deleting something. You stay in control; nothing sensitive happens silently. You can pre-approve categories of safe actions to reduce interruptions.

Sandbox. An optional protective boundary that restricts what files and network the session can touch, limiting the damage any mistaken or malicious action could do. Where available it is worth turning on. (In BCG's guidance, using a sandbox and a dedicated project folder is strongly encouraged.)

The tools available to Claude Code

These are the core capabilities Claude Code can use. You do not invoke them yourself — you speak in plain English and Claude chooses which to use — but recognising them makes its actions legible, and you will see their names as it works.

Sub-agent. A second (or third, or tenth) instance of Claude that the main session launches to work on a bounded task — "search the whole codebase for X", "review this file" — in its own separate context, returning just a summary. This keeps the main conversation uncluttered and lets independent jobs run at the same time. Think of it as delegating to a colleague and getting back a report rather than watching over their shoulder.

Extending Claude: skills, commands, memory, and MCP

Slash command. A shortcut you type starting with / that triggers a predefined action or workflow — for example /review to review changes, or /init to document a project. They are named starting-points for common tasks.

Skill. A packaged bundle of instructions and know-how that teaches Claude to do a particular kind of task well — formatting a specific document type, following a company procedure, using a specialised workflow. When a task matches, Claude loads the relevant skill and follows its guidance. Skills are how expertise gets reused rather than re-explained each time.

CLAUDE.md / memory. CLAUDE.md is a special plain-text file you can put in a project folder containing standing instructions, conventions, and facts about that project. Claude Code reads it automatically so you do not have to repeat yourself. More broadly, "memory" refers to notes Claude keeps so useful facts persist across sessions. (Security note: because Claude reads these files automatically, a CLAUDE.md inside a repository you did not write could contain instructions meant to manipulate the AI — so inspect such files in unfamiliar projects.)

MCP (Model Context Protocol). A protocol is an agreed set of rules for two systems to talk to each other. MCP is an open standard that lets Claude connect to external tools and data sources — your email, a calendar, a database, a company system, a design tool — through a common "plug". Before MCP, every integration was bespoke; MCP is like inventing a universal socket so any compatible device plugs in the same way.

MCP server. The small program that exposes a particular external system to Claude over MCP. There is a Gmail MCP server, a Slack one, a databases one, and so on. When you "connect an MCP server", you are giving Claude a new set of tools for that system. Its tools then appear alongside the built-in ones and are used the same way.

How MCP connects Claude to the outside world:

   +-----------+     MCP      +--------------+       +-------------------+
   |  Claude   | <==========> |  MCP server  | <===> |  External system  |
   |  Code     |  (standard   | (translator) |       |  (Gmail, Slack,   |
   |           |   protocol)  |              |       |   a database...)  |
   +-----------+              +--------------+       +-------------------+

   Claude speaks one standard language (MCP); each server translates it
   into whatever the specific external system understands.

A safety word about MCP servers and unfamiliar code: Only connect MCP servers your organisation has approved, and only run Claude Code against code you trust. Because the AI reads files and can run commands, a malicious project could try to smuggle instructions to it (a "prompt injection"). This is not a reason for fear — it is the same caution you would apply to opening an unknown email attachment.

Part 5 — Programming Languages and Python

Now to the material Claude Code most often works on: code itself.

What a programming language is

Programming language. A precise, written language for giving a computer unambiguous instructions. Unlike English, it has no room for vagueness — every symbol matters. There are many languages, each with strengths: Python for data and automation, JavaScript for the web, and so on.

Syntax. The grammar and punctuation rules of a language — where the brackets, colons, and indentation go. A syntax error means you broke a grammar rule (a missing bracket, say) and the computer refuses to proceed until it is fixed. Computers are utterly unforgiving about syntax.

Source code. The human-readable text a programmer writes, before the computer turns it into action. "The code" and "source" both refer to this.

Compiler vs interpreter. Computers ultimately run only low-level machine instructions, so source code must be translated. A compiler translates the whole program up front into a standalone runnable file. An interpreter translates and runs the code line by line, on the spot. Python uses an interpreter, which is part of why it is quick to experiment with — you can run a single line and see the result immediately.

Python

Python. A hugely popular programming language, prized for reading almost like English and for its enormous ecosystem of ready-made tools. It is the default language for data analysis, automation, scientific computing, and AI — and an excellent first language. Files end in .py. (The name comes from Monty Python, not the snake.)

A tiny taste, so the words below have something to point at:

   price = 100          # a VARIABLE holding a value
   tax = 0.2            # another variable

   def total(p, t):     # a FUNCTION named "total" taking two inputs
       return p * (1 + t)

   for item in [10, 20, 30]:      # a LOOP: repeat for each item
       print(total(item, tax))    # call the function, print the result

Variable. A named container that holds a value (a number, some text, a list). price = 100 stores 100 under the name price so you can refer to it later. The value can change — hence "variable".

Data type. The kind of value: a whole number (integer), a decimal (float), text (a "string"), a true/false (boolean), a list, and so on. Many bugs come from mixing types up, such as treating the text "100" as the number 100.

Function. A named, reusable block of instructions that takes some inputs and produces an output. Define it once, use it many times. total(...) above computes a price with tax. Functions are how programs avoid repetition and stay organised.

Loop. An instruction to repeat something for each item in a collection, or until a condition is met — the computer doing the tedious repetition for you.

Conditional (if / else). A branch: "if this is true, do X, otherwise do Y." How programs make decisions.

Comment. Text in the code that the computer ignores entirely — it exists purely to explain things to humans. In Python a comment starts with #. (Every language has its own comment symbol.) Good comments explain why something is done, not just what. Note this is one of two meanings of "comment" you will meet — the other is a remark left on a pull request, in Part 8.)

Reusing other people's code

Library / package / module. Rather than writing everything yourself, you import pre-built code others have written. A module is a single file of reusable code; a package or library is a collection of modules bundled together for a purpose (e.g. pandas for data tables, requests for fetching web pages). These three words are used loosely and often interchangeably; the key idea is "reusable code you bring in rather than write."

Standard library. The large set of packages that come built into Python already — dates, maths, file handling — no installation needed.

Import. The line of code (import pandas) that pulls a library into your program so you can use it. Installing a package (Part 6) puts it on your machine; importing it brings it into a specific program.

When things go wrong

Bug. A mistake in code that makes it behave wrongly. Debugging is the detective work of finding and fixing bugs — a large share of all programming.

Error / exception. When code hits something it cannot handle, it stops and reports an error (in Python, an "exception"), usually with a message and a traceback — a printout of exactly where it failed. Error messages look intimidating but are your best clue; reading the last line first is a good habit, and pasting the whole thing to Claude is a great way to get unstuck.

Part 6 — Environments, Packages, PATH, Node and npm

Code does not run in a vacuum. It runs inside an environment — and understanding that word dissolves a whole cluster of confusions (including the second meaning of "path").

The environment

Environment. The complete surroundings in which a program runs: which version of the language is installed, which packages are available, and various settings. Two computers — or two projects on one computer — can have different environments, which is why "it works on my machine" is a famous programmer's lament. Much of the setup work in software is about getting the environment right.

Runtime. The software that actually executes a program of a given language — the Python runtime runs Python, the Node runtime runs JavaScript. "At runtime" also just means "while the program is running" (as opposed to while you are writing it).

Environment variable. A named setting that lives in the environment, outside your code, which programs can read. It is used for configuration — things like where to find a tool, or a secret API key — so the same code can behave differently in different environments without editing the code. (Best practice: keep secrets such as passwords and keys in environment variables, never written into your code files.)

PATH (the environment variable). The single most important environment variable, and the second meaning of "path" in this document. PATH is a list of folders the shell searches, in order, whenever you type a command, so it knows where to find the corresponding program. When an install guide says "add it to your PATH", it means "put this program's folder in that list so I can run it by name from anywhere." The dreaded command not found error usually means the program exists but its folder is not on your PATH.

You type:   python

Shell checks each folder on PATH, in order, for a program called "python":
   C:\Windows\System32        -> not here
   C:\Python312\              -> found it!  runs this one
   C:\Users\...\some\folder   -> (never reached)

Two senses of "path", side by side:
   * a PATH to a file      = the address of one file        (Part 1)
   * the PATH variable     = a list of folders to search    (this part)

Getting and managing packages

Dependency. A package your project relies on to work. If your program imports pandas, then pandas is a dependency — it must be present in the environment or the program breaks. Real projects often have dozens, which in turn depend on others; managing this web is a job in itself.

Package manager. A tool that downloads, installs, updates, and removes packages for you, sorting out the web of dependencies automatically. Every ecosystem has one.

pip. Python's package manager. pip install pandas fetches the pandas package from the Python Package Index (PyPI, the central online repository) and installs it into your environment. This is how you get the tens of thousands of Python libraries onto your machine.

Node and npm (the JavaScript world)

Node.js. A runtime that lets JavaScript — originally a language that only ran inside web browsers — run anywhere, including on servers and your own command line. Many developer tools (Claude Code and the Google Workspace CLI among them) are built on Node. Installing Node is often a prerequisite for other tools. (Note: "Node.js" is unrelated to the general word "node" — see Part 11.)

npm (Node Package Manager). Node's package manager — the JavaScript world's equivalent of pip. npm install <name> fetches and installs a JavaScript package from the npm registry (the huge central store of JS packages). When you install a Node-based tool "globally" (npm install -g), it becomes available as a command everywhere on your machine.

npx. A companion to npm that runs a package's command without permanently installing it — handy for one-off use of a tool.

Keeping projects apart

Virtual environment (venv). An isolated, self-contained Python environment for a single project, with its own private set of installed packages. Without it, every project shares one global set, and Project A needing an old version of a library while Project B needs a new one causes painful clashes. A virtual environment gives each project its own clean room. You "activate" it while working on that project. (Node achieves similar isolation by installing packages into a per-project node_modules folder.)

requirements.txt / package.json. A plain-text file listing exactly which packages (and often which versions) a project needs. Anyone — including a future you, or a colleague, or Claude Code — can recreate the right environment from this file with one command. It is how projects stay reproducible.

Part 7 — Jupyter Notebooks

A Jupyter Notebook is one of the most popular ways to write Python, especially for data analysis and learning — and a great environment for working with Claude.

Jupyter Notebook. An interactive document that mixes runnable code, its output, and formatted notes in one place. Instead of writing a whole program and running it at the end, you build it up in small pieces and run each piece immediately, seeing results (including charts and tables) inline. The file ends in .ipynb. It runs in your web browser but the file lives on your computer.

Cell. A notebook is a stack of cells — individual blocks you run one at a time. There are two main kinds:

Code cell — contains code; when you run it, the output appears directly beneath.
Markdown cell — contains formatted text (headings, notes, explanations) using Markdown (Part 12); when you run it, it renders as tidy prose. This is how notebooks double as readable reports.

Kernel. The behind-the-scenes engine that actually runs your code and remembers everything defined so far in the session (your variables, imported libraries, and so on). If a notebook starts misbehaving in confusing ways, "restarting the kernel" wipes that memory and gives you a clean slate — a very common fix.

Output. Whatever a code cell produces when run — a number, a table, a chart, an error — displayed beneath the cell and saved inside the notebook file.

A notebook, top to bottom:

   +-------------------------------------------------+
   | [Markdown cell]  # Sales Analysis               |  <- renders as a heading
   +-------------------------------------------------+
   | [Code cell]      import pandas as pd            |  <- run this...
   |                  data = pd.read_csv("sales.csv") |
   +-------------------------------------------------+
   | [Code cell]      data.head()                    |  <- ...then this
   |  Output:         (a table of the first 5 rows)  |  <- output appears here
   +-------------------------------------------------+

   The KERNEL remembers "data" from the second cell when it runs the third.
   Run order matters: cells share memory, so run them top to bottom.

Run order gotcha. Because all code cells share the kernel's memory, running them out of order — or re-running an earlier one after changing things — can produce confusing states. A good habit for a finished notebook is "Restart kernel and run all" to confirm it works cleanly from top to bottom.

Part 8 — Version Control with Git

This part contains a dense cluster of the terms you asked about — git, committed, main, worktree, pull requests, comments — because they all belong to one system. Learn the system and the words fall into place.

The problem it solves

Version control. A system that records the complete history of changes to a set of files, so you can see what changed, when, and by whom — and rewind to any earlier state. It replaces the chaos of report_final_v2_FINAL_really.docx with one clean, fully-tracked history. It is also what lets several people work on the same project without overwriting each other.

Git. By far the most widely used version-control system. It runs on your machine, tracks a project's history, and is operated mainly through the command line (Part 2). Git is the tool; the terms below are its vocabulary. (Git is not the same as GitHub — see below.)

The core objects

Repository ("repo"). A project folder that Git is tracking. It contains your files plus a hidden .git subfolder holding the entire history. "Cloning a repo" means downloading a copy of one, history and all.

Working tree / working directory. The actual files as they exist right now in the folder — the version you see and edit. This is the everyday, editable state, as opposed to the saved history tucked away in .git. (The word "worktree" you asked about is this — plus an advanced feature below.)

Commit. A saved snapshot of the project at one moment, with a short message describing what changed. When something is committed, it is permanently recorded in the history and can always be returned to. Commits are the fundamental unit of Git — the history is simply a chain of them. Good practice is small, frequent commits, each with a clear message ("fix login validation") so the history reads like a logbook.

Staging area (the "index"). A holding area between your working files and a commit. You stage the specific changes you want (git add), then commit them (git commit). This two-step lets you commit a tidy, deliberate set of changes rather than everything at once.

The journey of a change in Git:

  WORKING TREE        STAGING AREA        LOCAL REPO           REMOTE
  (files you edit)    (the "index")     (.git history)     (e.g. GitHub)
        |                  |                  |                  |
        |   git add        |   git commit     |   git push       |
        +----------------> +----------------> +----------------> +
                                              |                  |
                                              | <--- git pull ----+
                                              |  (bring down others'
                                              |   committed changes)

Branches and where you are

Branch. A parallel line of development. You split off a branch to work on a feature or fix in isolation, without disturbing the main version; when it is ready, you merge it back. Branches are cheap and central to how teams collaborate — everyone works on their own without stepping on each other.

main (or master). The conventional name of the primary branch — the definitive, official version of the project. Historically called master; the modern default is main. Work usually happens on side branches and is merged into main once reviewed and tested, keeping main stable and trustworthy.

HEAD. Git's pointer to "where you are right now" — usually the tip of the branch you currently have checked out. You rarely think about it directly, but you will see the word.

Checkout / switch. Moving from one branch (or past commit) to another, which swaps the files in your working tree to match. "Check out the main branch" means "make my working files show main's version."

Worktree (the advanced feature). Normally a repo has one working tree, and switching branches swaps the files in place. Git's worktree feature lets you check out several branches at once, each in its own separate folder, all backed by the same repository history. That way you can have main open in one folder and a feature branch in another simultaneously — useful when you want to work on two branches without constantly switching. (This is why you may see Claude Code offer to work "in a worktree": it gives the AI an isolated copy of a branch to work in without disturbing your current files.)

Working with others: remotes and GitHub

Remote. A copy of the repository hosted elsewhere — typically on a server — that you sync with. It is the shared central copy the team gathers around.

origin. The conventional name for your project's main remote. "Push to origin" means "send my commits to the shared central copy."

GitHub / GitLab / Bitbucket. Websites that host Git remotes and add collaboration features on top — a web view of the code, issue tracking, code review, and automation. Crucial distinction: Git is the underlying tool that runs on your machine; GitHub is a popular website for hosting Git repositories and working on them together. You can use Git with no GitHub at all.

Clone. Download a full copy of a remote repository to your machine, history included, to start working on it.

Fork. Your own personal copy of someone else's repository on GitHub, which you can change freely — the usual starting point for contributing to a project you do not own.

The everyday sync verbs

VerbMeaning
pushUpload your local commits to the remote, sharing them with everyone.
pullDownload and merge in everyone else's latest commits from the remote.
fetchDownload others' commits to look at them, without merging into your work yet.
mergeCombine the changes from one branch into another.

Diff. A line-by-line display of exactly what changed between two versions — additions in one colour, deletions in another. Reading diffs is how you (and reviewers, and Claude Code) see precisely what a change does. "The diff" often just means "the set of changes."

Merge conflict. When two people change the same lines in incompatible ways, Git cannot decide automatically and asks a human to resolve it — you edit the file to choose the correct combination. Conflicts sound alarming but are a normal, routine part of collaboration.

.gitignore. A file listing things Git should not track — temporary files, huge data files, and especially secrets. This is how you keep passwords and API keys out of your history.

Pull requests and code review

Pull request (PR). A GitHub feature (called a "merge request" on GitLab) for proposing that a branch's changes be merged into another branch, usually main. Opening a PR says "here is my finished work — please review it before it becomes official." It shows the diff, lets others comment, runs automated checks, and provides the button to merge once approved. The pull request is the hub of modern team development: almost no change reaches main without going through one.

Code review. The practice of having others read a proposed change (on the PR) before it is merged — catching bugs, suggesting improvements, and spreading knowledge. AI-generated code should go through the same review as any other.

Comment (review sense). A remark left on a pull request — a question, a suggestion, an approval — often attached to a specific line of the diff. This is the second meaning of "comment": in Part 5 a comment was an ignored note inside the code; here it is human feedback on a proposed change. Same word, two settings.

Merge (a PR). The act of accepting a pull request, folding its changes into the target branch. After merging, the change is part of the official project.

The pull-request lifecycle:

   1. Branch off main         main o---o---o
                                          \
   2. Commit your work                     o---o   (your feature branch)
                                                \
   3. Open a Pull Request  --> [review + comments + automated checks]
                                                \
   4. Merge when approved     main o---o---o-----o   (changes now official)

Part 9 — The Web: HTML, Websites, and Hosting

One of your goals is building and hosting HTML websites. This part gives you the whole vocabulary, from what a page is made of to how it reaches the world.

What a web page is made of

Every website is built from three languages, each with a distinct job:

HTML (HyperText Markup Language). The structure and content of a page — the headings, paragraphs, lists, links, and images, and how they are nested. HTML is not really "programming"; it is markup: you wrap content in tags to label what each part is. A file ends in .html, and every website begins here. (This very document was written in HTML before being converted to a Google Doc.)

CSS (Cascading Style Sheets). The appearance — colours, fonts, spacing, layout. HTML says "this is a heading"; CSS says "headings are dark blue, 24 points, centered." Separating content (HTML) from styling (CSS) is a core web principle.

JavaScript (JS). The behaviour — the interactivity that happens in your browser: buttons that respond, content that updates without reloading, animations, forms that validate. (Reminder: JavaScript the language is what Node.js, from Part 6, lets you run outside the browser too.)

The three web languages, by role:

   HTML  ->  structure & content   ("what is on the page")
   CSS   ->  presentation & style  ("what it looks like")
   JS    ->  behaviour & logic     ("what it does when you interact")

   A helpful analogy for a house:
   HTML = the walls and rooms   CSS = the paint and furniture   JS = the electrics

Tag / element. HTML's building block. A tag like <p> opens a paragraph and </p> closes it; the opening tag, closing tag, and content between form an element. Tags nest inside one another to build the page's structure.

Browser. The program that fetches web files and renders them — turns the HTML, CSS, and JS into the visual page you see. Chrome, Safari, Edge, Firefox. "Rendering" is the act of drawing the page from its code.

Client and server

Client and server. The web runs on a request-and-response conversation. The client (your browser) requests a page; a server (a computer that stores the site and is always on) responds by sending the files back. This client-server split underlies almost everything online.

   BROWSER (client)                                 SERVER
   +-------------+     1. request: "send me         +-------------+
   |             | -------- example.com" ---------> |             |
   |  You, on    |                                  |  Stores the |
   |  your laptop| <----- 2. response: the -------- |  website's  |
   |             |        HTML, CSS, JS files       |  files      |
   +-------------+                                  +-------------+
        |
        v  browser renders the files into the visible page

Frontend and backend. Two halves of a web application:

TermWhere it runsWhat it handles
FrontendIn the browser (client-side)Everything the user sees and touches — HTML, CSS, JS. Also called "client-side".
BackendOn the server (server-side)The behind-the-scenes engine — databases, logic, accounts, saved data. The user never sees it directly. Also called "server-side".

A simple HTML website may be frontend-only. A web app like online banking has a rich backend too. Full-stack means working across both.

How pages are addressed and delivered

HTTP / HTTPS. The rules (the protocol) browsers and servers use to exchange requests and responses. The S in HTTPS means secure — the traffic is encrypted so others cannot read it in transit. Always prefer HTTPS; browsers now flag plain HTTP as "not secure".

URL. A web address, e.g. https://www.example.com/about. It names the protocol (https), the domain (example.com), and the path to the specific page (/about) — yes, "path" again, now inside a web address.

Domain / domain name. The human-friendly name of a site, like example.com, which you register (rent) so people can find you. DNS (Domain Name System) is the internet's phone book: it translates that name into the numeric address of the server behind it.

IP address. The actual numeric address of a computer on a network, like 142.250.72.14. DNS exists so you can type memorable names instead of numbers.

localhost / port. localhost means "this very computer" — when developing a website you run it locally and view it at localhost before the world can see it. A port is a numbered channel on a machine so several programs can use the network at once; you will often see addresses like localhost:8000, where 8000 is the port your local site is served on.

Getting a site onto the internet

Hosting. Putting your website's files on a server that is permanently connected to the internet, so anyone can reach them. A web host is a company that provides that always-on server (for a fee, or sometimes free for simple sites). Your laptop is not a host — it is not always on and not built for it.

Static vs dynamic site. A static site serves the same pre-written files to everyone (a personal site, a blog, a brochure) — simple, fast, cheap, and often free to host. A dynamic site builds pages on the fly per user, needing a backend (think a logged-in dashboard). For your HTML-website goal, static hosting is the sweet spot to start.

Deploy / deployment. The act of publishing your site (or app) to its host so the new version goes live. "Deploying" a static site can be as simple as pushing to a Git repository that the host watches.

Static site hosts. Popular, beginner-friendly places to host static HTML sites, several free: GitHub Pages (serves a site straight from a GitHub repository), Netlify, Vercel, and Cloudflare Pages. A common modern workflow: keep the site in a Git repo, connect it to one of these, and every push automatically deploys the update.

Part 10 — The Cloud, Data, S3, and APIs

Where large-scale storage, data, and services live — including the S3 term you asked about.

The cloud

The cloud / cloud computing. Using computing resources — storage, servers, databases — that run in someone else's data centre and are rented over the internet, instead of owning the hardware yourself. "The cloud" is not mystical: it is just other people's computers, professionally run, that you pay for as you use them. The big three providers are AWS (Amazon Web Services), Microsoft Azure, and Google Cloud (GCP).

AWS (Amazon Web Services). Amazon's cloud platform and the market leader — a huge menu of rentable services, of which S3 is one of the best known.

S3 and object storage

S3 (Simple Storage Service). AWS's massively scalable file-storage service — where a great deal of the world's data physically sits. When people say "the data is in S3" or "an S3 bucket", they mean files stored in this service. It is built to hold anything from a single file to trillions, reliably and cheaply, reachable over the internet by programs.

Bucket. A top-level container in S3, like a giant labelled folder, with a globally-unique name. Your files live inside buckets.

Object / key. In S3, each stored file is an object, and its full name (which looks path-like, e.g. reports/2026/q1.csv) is its key. S3 has no real folders — the slashes in a key just look like folders for convenience. This is why S3 is called "object storage" rather than a filesystem.

   AWS S3
    │
    ├─ Bucket: "acme-company-data"          (globally unique name)
    │     ├─ Object, key = reports/2026/q1.csv
    │     ├─ Object, key = reports/2026/q2.csv
    │     └─ Object, key = images/logo.png
    │
    └─ Bucket: "acme-website-backups"
          └─ Object, key = 2026-07-15/site.zip

Databases

Database. An organised store of data designed for fast, reliable searching, updating, and retrieval at scale — far beyond what a spreadsheet handles. Almost every app has one behind it, in the backend.

SQL vs NoSQL. SQL (Structured Query Language) databases store data in linked tables of rows and columns, like related spreadsheets, and you query them with the SQL language (SELECT name FROM customers WHERE ...). NoSQL databases use more flexible structures for cases where rigid tables do not fit. SQL is the safe default assumption when someone says "the database".

APIs: how programs talk to each other

API (Application Programming Interface). A defined way for one program to request something from another, without knowing its inner workings. When your code asks a weather service for today's forecast, or Claude connects to Gmail, it uses that service's API — a published set of requests you are allowed to make and the responses you get back. An API is a contract: "send a request shaped like this, get a result shaped like that." (An MCP server, from Part 4, is essentially a wrapper that turns an external system's API into tools Claude can use.)

Endpoint. A specific URL you send an API request to, corresponding to one operation — e.g. one endpoint to fetch a user, another to create one.

REST. The most common style/convention for web APIs, using ordinary web addresses and HTTP verbs (get, create, update, delete). "A REST API" just means an API that follows these familiar conventions.

API key / token. A secret string that identifies and authorises you to an API — like a password for your program. It should be kept out of your code (use environment variables, Part 6) and never committed to Git or shared.

How data is written down

JSON (JavaScript Object Notation). The near-universal text format for exchanging structured data between programs — almost every API speaks JSON. It represents data as nested keys and values, and is readable by both humans and machines. You have seen it throughout this document whenever a tool took {...} input.

   A small JSON example:

   {
     "name": "Benjamin",
     "role": "consultant",
     "projects": ["glossary", "website"],
     "active": true
   }

   Curly braces {} group related fields; each is  "key": value.
   Values can be text, numbers, true/false, lists [ ], or more nested objects.

CSV (Comma-Separated Values). The simplest data format: plain-text rows, with columns separated by commas — effectively a spreadsheet as text. Opens in Excel, reads easily in Python. Ubiquitous for tabular data.

YAML. Another human-friendly text format, common for configuration files, where structure is shown by indentation rather than braces. You will meet it in tool and project settings.

Part 11 — "Node" and Other Words That Mean Several Things

A few words are genuinely overloaded — they mean different things in different contexts, which is a common source of confusion. You asked specifically about "nodes"; here it is, with its cousins.

Node (the general concept). At its most general, a node is a single point or unit within a larger connected structure — one dot in a network of connected dots. That one idea shows up in several specific guises:

"Node" in this context......means

A tree or graph nodeOne item in a branching data structure — e.g. a folder in the filesystem tree, or an element in HTML. Each has a parent and possibly children.
A compute / cluster nodeOne individual computer within a larger group of machines working together (a "cluster"), as in cloud computing or big-data processing.
A network nodeAny device connected to a network — a computer, phone, or server — that can send or receive data.
A node in your memory filesYou may notice node_type: memory in Claude's memory notes — here a node is one stored unit of information in a connected knowledge structure.

How to tell which is meant: look at the surrounding words. "Install Node" or "npm" → Node.js. "Parent/child node" → a tree structure. "A node in the cluster" → a computer. The underlying image — a point in a connected structure — is the thread linking them all.

Part 12 — Tools of the Trade and Putting It Together

The remaining vocabulary of the day-to-day working environment, and how the pieces fit into projects.

Where code gets written

IDE (Integrated Development Environment). A single application that bundles everything for writing code — a text editor, a way to run programs, debugging tools, Git integration, and a built-in terminal. "Integrated" because it puts the whole toolkit in one window.

VS Code (Visual Studio Code). The most popular IDE/editor by far — free, from Microsoft, extendable, and the usual home for Python, web work, and Claude Code's editor integration. (Not to be confused with the older, heavier "Visual Studio".)

Extension / plugin. An add-on that gives an editor or tool new abilities — language support, themes, integrations. You install the ones you need; they are how VS Code adapts to any kind of work.

Administrator / root / sudo. The elevated permission level that can change anything on a computer, including system files. On Mac/Linux you temporarily borrow it with sudo; the all-powerful account is called root. Powerful and therefore risky — only use elevated privileges when genuinely required and when you understand the command, since mistakes at this level can damage the system.

Files that describe and configure projects

Config(uration) file. A file holding a program's settings, separate from its code, so behaviour can change without editing the program. Common formats are JSON, YAML, and TOML. Editing config files is a routine part of setting tools up.

Markdown. A lightweight way to write formatted text using plain characters: # for a heading, **bold**, - for bullet points. It stays readable as plain text yet renders into tidy formatting. It is everywhere — README files, notebook text cells, chat, and documentation. Files end in .md. (CLAUDE.md from Part 4 is a Markdown file.)

README. The front-door document of a project (usually README.md), explaining what it is, how to set it up, and how to use it. The first thing to read when you open an unfamiliar repository — and the first thing you should write when you make one.

Project. Informally, a self-contained folder holding everything for one piece of work — its code, its data, its config, its README, often its own Git repository and virtual environment. "Working in a project" means keeping all of that together and tidy in one place, which is exactly the habit that makes tools like Claude Code work smoothly (point it at the project folder and it has everything it needs).

A few more you will bump into

Open source. Software whose source code is published for anyone to read, use, and contribute to, usually free. Python, Git, VS Code, and countless libraries are open source. A licence is the legal text saying what you may do with it.

Regular expression (regex). A compact pattern-language for matching text — e.g. "find every email address." Powerful, famously cryptic, and something Claude is very good at writing for you so you do not have to memorise the symbols.

Framework vs library. A library is code you call when you want it; a framework is a larger structure that calls your code, giving a ready-made skeleton for a whole application (e.g. a web framework). Loosely: a library is a tool in your hand; a framework is the workshop you build inside.

Terminal multitasking — background vs foreground. A command that runs and returns control to you immediately is running in the foreground; a long-running one can be sent to the background so you keep working while it churns. Claude Code uses this to run slow jobs without blocking.

A suggested learning path

You do not need to learn all of this at once. Given your goals, a sensible order:

1. Get comfortable on the command line (Parts 1–2). Practise moving around folders with cd, ls, pwd. Everything else sits on this. Ask Claude Code to explain any command before you run it.

2. Learn just enough Python (Part 5), in a Jupyter Notebook (Part 7). Notebooks give instant feedback, which is the fastest way to learn. Start with variables, printing, and simple loops. Have Claude generate small examples and explain them line by line.

3. Understand environments and packages (Part 6) the first time you hit "module not found" — that error is your cue to learn pip and virtual environments. This is where most beginners get stuck, so expect it and do not be discouraged.

4. Build a small HTML page (Part 9) — even one file — then host it free on GitHub Pages. Doing this once makes "frontend", "hosting", and "deploy" concrete forever.

5. Put a project under Git (Part 8) once you have something worth keeping. Learn commit first; add branches and pull requests later, when collaborating.

6. Lean on Claude Code throughout (Part 4). Point it at a project folder, describe what you want in plain English, and read what it proposes before approving. Treat its output as a skilled draft to review and test — never as automatically correct. Paste error messages straight to it; explaining errors is one of its strengths.
